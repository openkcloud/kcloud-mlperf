# `scripts/publish/` — WS-H Publication Bundle

This directory ships the WS-H **Publication Bundle (MLPerf 5.1 Compliant)**
toolchain. It builds a self-contained, reviewer-ready tarball from one or
more benchmark runs in the comparison API.

Per AG-10 = **(a)**, the bundle aims to mirror the MLPerf-Inference v5.1
submission layout as closely as possible. Where source runs were not
executed via the upstream LoadGen harness, the bundle emits clearly
labelled placeholder files and appends a caveat to
`methodology_caveats.md`.

## Files in this directory

| File                                | Purpose                                                           |
| ----------------------------------- | ----------------------------------------------------------------- |
| `build-publication-bundle.ts`       | TypeScript CLI that builds the bundle tarball.                    |
| `submission_checker_local.py`       | Python pre-flight that validates the bundle structure (subset of upstream `submission_checker.py`). |
| `README.md`                         | You are here.                                                     |

The dual-checklist audit template lives separately at
`docs/publication-bundle/methodology_audit_template.md`.

## Quick start

```bash
# Print help
npx ts-node \
  --compiler-options '{"module":"commonjs","moduleResolution":"node","target":"es2020","esModuleInterop":true,"skipLibCheck":true}' \
  scripts/publish/build-publication-bundle.ts --help

# Dry-run to see the manifest only (no tarball written)
npx ts-node \
  --compiler-options '{"module":"commonjs","moduleResolution":"node","target":"es2020","esModuleInterop":true,"skipLibCheck":true}' \
  scripts/publish/build-publication-bundle.ts --all-canonical --dry-run

# Build the canonical-only bundle against a live backend
npx ts-node \
  --compiler-options '{"module":"commonjs","moduleResolution":"node","target":"es2020","esModuleInterop":true,"skipLibCheck":true}' \
  scripts/publish/build-publication-bundle.ts \
  --all-canonical --api http://localhost:9999 \
  --out dist/publication-bundles

# Build for a single sweep id
npx ts-node \
  --compiler-options '{"module":"commonjs","moduleResolution":"node","target":"es2020","esModuleInterop":true,"skipLibCheck":true}' \
  scripts/publish/build-publication-bundle.ts \
  --sweep S2026-05-11-llama8b

# Build for a single exam id
npx ts-node \
  --compiler-options '{"module":"commonjs","moduleResolution":"node","target":"es2020","esModuleInterop":true,"skipLibCheck":true}' \
  scripts/publish/build-publication-bundle.ts --exam 1234
```

The compiler-options blob exists because the repo `tsconfig.json` is
NestJS-style (`module: nodenext`) and ts-node needs an explicit CJS
override when invoked from outside `nest start`. If you build the script
via `tsc` first you can drop it.

## Bundle structure

```
<bundle-id>/
├── ADR.md                                  # decision record for sweep parameters
├── manifest.json                           # generated by the script — list of files + caveats
├── methods.md                              # auto-generated from exam metadata
├── methodology_caveats.md                  # incompatibility reasons + warmup/cache notes
├── reproducibility.json                    # N=11 metadata per row + chart_sha256
├── results.csv                             # denormalized table (full canonical only)
├── compliance/                             # placeholder TEST01/04/05 stubs
│   ├── TEST01/verify_accuracy.txt
│   ├── TEST04/verify_performance.txt
│   └── TEST05/verify_performance.txt
├── mlperf/                                 # one subdir per MLPerf run id
│   └── <run-id>/
│       ├── accuracy.txt                    # AccuracyMode placeholder
│       ├── mlperf_log_detail.txt           # MLLOG placeholder
│       ├── mlperf_log_summary.txt          # summary placeholder
│       ├── mlperf.conf                     # mlperf.conf placeholder
│       ├── system_desc_id.json             # SUT description (valid JSON)
│       └── user.conf                       # user.conf placeholder
└── raw_logs/                               # one stub file per run, listing artifact paths
    └── run-<id>.log
```

The tarball is written to
`<out-dir>/<bundle-id>.tgz` where `<bundle-id>` is
`{exam|sweep|all-canonical}[-<id>]-<built-at-iso>` with `:` and `.`
swapped for `-` so the filename is portable.

## `submission_checker_local.py`

A lightweight pre-flight that verifies bundle **structure** — it does
**not** validate LoadGen log contents, accuracy thresholds, or compliance
verdicts.

```bash
python3 scripts/publish/submission_checker_local.py dist/publication-bundles/all-canonical-2026-05-11T03-04-05-000Z.tgz
python3 scripts/publish/submission_checker_local.py path/to/extracted/bundle-dir/
```

Exit codes:

- `0` — every required artifact present + JSON files parse.
- `1` — one or more required artifacts missing or invalid.
- `2` — bundle / tarball not found or unreadable.

> **For a real MLPerf 5.1 submission**, after this local check passes,
> you MUST clone `mlcommons/inference` and run
> `tools/submission/submission_checker.py` against the same directory.
> Upstream additionally validates LoadGen log integrity, accuracy
> thresholds, scenario rules, and the compliance suite verdicts — all of
> which are **out of scope** for the local checker.

## DoD #8 dual-gate user-confirmation procedure

Per the WS-H mega-plan Definition of Done, every published bundle must
pass two human gates before release:

1. **Author gate**: the run owner fills in
   `docs/publication-bundle/methodology_audit_template.md` (copied
   alongside the bundle) and marks every Pineau / MLPerf checklist item
   `Y / N / N-A` with a comment.
2. **Reviewer gate**: a second engineer (not the run owner) verifies the
   audit answers against the bundle contents, runs
   `submission_checker_local.py`, and signs off in Section C of the
   audit template.

A bundle ships only when **both** gates show `accept`.

## Known gaps

These are deliberate scope boundaries for WS-H, documented here so users
don't expect more than the bundle delivers.

- **LoadGen output**: real `mlperf_log_summary.txt`, `mlperf_log_detail.txt`,
  and `accuracy.txt` are produced by the upstream MLPerf LoadGen harness
  during the run — they cannot be reconstructed after the fact. ETRI
  benchmark runs that did *not* go through LoadGen ship placeholder
  files; a caveat is added to `methodology_caveats.md`. Closing this gap
  requires modifying the per-vendor benchmark harness (RNGD / Atom+ /
  GPU sweep) to invoke LoadGen in `--scenario Offline` mode, which is
  out of scope for WS-H.
- **Compliance suite results**: `compliance/TEST01,TEST04,TEST05/` ship
  placeholders for the same reason — the suite must be re-run against
  LoadGen output before real values can be emitted.
- **Real `submission_checker.py`**: the local checker is a strict
  subset. For a real submission, run upstream
  `mlcommons/inference/tools/submission/submission_checker.py`. The
  local script is only a fast pre-flight.
- **Raw logs**: per-pod logs are a best-effort placeholder. The bundle
  builder records the candidate NFS / Loki paths but does not actually
  shell out to `kubectl cp` or query Loki — keeping the script
  side-effect free. Operators can populate `raw_logs/run-<id>.log`
  manually before tar'ing.

## Snapshot test

The structure contract is enforced by
`server/test/publication-bundle.spec.ts`, which feeds a fixture
`ListResponse` into the script via the injectable `fetchJson` hook,
asserts the manifest matches the expected file set, and (when `tar` is
available) untars the output and re-checks every entry. The test is
plain Jest:

```bash
cd server
npx jest --config test/jest-e2e.json test/publication-bundle.spec.ts
```

The test mocks the upstream API entirely and never touches the cluster.
A live-cluster smoke test is intentionally deferred until the WS-G live
dashboard is wired in.
